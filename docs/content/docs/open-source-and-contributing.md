# Open Source and Contributing

## Open source posture

This project should be easy to inspect, improve, and adapt. The code is readable, the documentation should explain real decisions, and contributions should keep the product feeling intentional rather than chaotic.

## Good contributions

Good changes usually do one of these well:

- improve documentation structure or clarity
- fix navigation and reading experience issues
- strengthen import, parsing, or publishing behavior
- reduce fragile state or undefined runtime behavior

## Contribution expectations

- keep changes focused
- preserve the single-workspace and category-based mental model
- test both editor behavior and published reading behavior
- avoid mixing unrelated refactors into one change set

## Documentation contributions

When adding docs, place them in the right category, use concise summaries, and write with real readers in mind. A short useful page is better than a long vague one.

## Design principle

The app should stay simple, but not careless. Developer docs need clarity more than feature overload.
